奥秘城市V2客户端接口文档

  • 通信协议：websocket，安卓客户端使用autobahn开源websocket通信库，服务器使用Spring(4.1.5) WebSocket API
  • 请求响应框架：服务器和客户端分别基于websocket构建的Request,Response请求响应框架。通信时双向的，服务器构建Request的处理框架，客户端构建Response的响应框架。
  1. 注册 register
  2. 登陆 login
  3. 加能量 addEnergy
  4. 更新头像 updateAvatar
  5. 刷新位置接口 freshLocation（暂定5秒为周期）
  6. 获取周边能量点可能出现的地点集合 getEnergyPoints
  7. 获取周边可接受的任务 getTaskView
  8. 接受地图上的任务 acceptTask
  9. 放弃任务 abortTask
  10. 消息界面答题 answerTask
  11. 装备道具接口 equipProperty
  12. 获取道具列表 listProperty
  13. 任务推送 taskPush 注：此接口不需要请求，服务器主动通知，客户端写好response回调即可
  14. 扫码 scanPrevRule
  15. 新任务在线推送接口 newTasks
  16. 已接受任务消息推送 taskMsg

1.注册 register
Request：
如果为游客开始游戏，下属字段置为空串

  • username，字符串，用户昵称

  • photo，字符串，头像url链接地址

  • openId，字符串，微信或QQ的openId

Response：
status返回200，注册成功

  • userId 整型，用户唯一id

    新注册的用户服务器会返回给客户端的用户标识码（就是数据库表中用户表的自增主键）
2.登陆接口，method: login
Request :

  • userId，整型，用户唯一id

    新注册的用户服务器会返回给客户端的用户标识码（就是数据库表中用户表的自增主键）

Response：
status返回100，用户不存在
status返回101，用户已登录
status返回200，登陆成功
  • username，字符串（需要url编码），用户昵称
  • photo，字符串，头像路径
  • score，整形，积分
  • energy，整形，能量值

  • coupons，整型，优惠券
  • level，整型，等级
  • upgradeGap，整型，升级所需的经验值

3.加能量，method: addEnergy
Request :

  • userId，整型，用户唯一id

  • energy，整型，能量值

    客户端自己从随机点集中产生能量点的位置，自己收集能量点，然后将收集的能量增加值同步给服务器

Response：
服务器不响应
4.更新头像，method: updateAvatar
Request :

  • userId，整型，用户唯一id

  • photoBase64，字符串，base64编码的图片

Response：
status返回200，修改成功
status返回500，修改失败（几率很小）
5.刷新位置接口，method: freshLocation（客户端自己定，暂定5秒为周期）
Request :

  • userId，整型，用户唯一id

  • lng，6位小数的浮点数，用户经度

  • lat，6位小数的浮点数，用户纬度

Response：
无需响应
6.获取周边能量点可能出现的地点集合 getEnergyPoints
Request :

  • userId，整型，用户唯一id

  • lng，6位小数的浮点数，用户经度

  • lat，6位小数的浮点数，用户纬度

Response：
status返回200
  • points 经纬度map数组，能量点集的经纬度
示例：
"points"：[
    {"lng":108.123456,"lat":34.132564},
    {"lng":108.123456,"lat":34.132564},
    ......
]
能量点可能出现地点的点集
7.获取周边可接受的任务 getTaskView
Request :

  • userId，整型，用户唯一id

  • lng，6位小数的浮点数，用户经度

  • lat，6位小数的浮点数，用户纬度

  • type，字符串，任务类型，有三种可选："ACCEPT","VIEW","DUMPED"，分别表示已接受，任务视图，已放弃的任务

Response：
status返回200
  • type，字符串，任务种类，值与request中的type一致，客户端据此判断任务类型
  • tasks，任务数组
tasks示例：
"tasks"：[
  {
    taskId，整型，任务id
    title，字符串，任务标题
    info，字符串，介绍

    difficulty，整型，任务难度

    clues，字符串，任务线索

    taskType，整型，1表示线上，0表示线下

    score，整型，积分
    lng，float型，任务地点经度
    lat，float型，任务地点纬度
},
{
     taskId，整型，任务id
    title，字符串，任务标题
    info，字符串，介绍

    difficulty，整型，任务难度

    clues，字符串，任务线索

    taskType，整型，1表示线上，0表示线下

    score，整型，积分
    lng，float型，任务地点经度
    lat，float型，任务地点纬度
}
    ......
]
8.接受地图上的任务 acceptTask
Request :

  • userId，整型，用户唯一id

  • taskId，整型，任务id

Response：
status返回200 处理成功
无数据
9.放弃任务 abortTask
Request :

  • userId，整型，用户唯一id

  • taskId，整型，任务id

Response：
status返回200 处理成功
无数据
10.消息界面答题 answerTask
Request :

  • userId，整型，用户唯一id

  • taskId，整型，解密任务id

  • content，字符串，输入的内容

Response：
status返回403 没有权限答题，即前置规则没有通过
status返回200 答题成功，但是答错还是答对都返回200，具体从反馈回复里看
  • reply，map数组，返回的html消息
示例：
"reply" : [
{"html":"html文本1"},
{"html":"html文本2"},
...
]
11.装备道具接口 equipProperty
Request :

  • userId，整型，用户唯一id

  • targetId，整型，要被替换的那个道具的id

  • replaceId，字符串，要装备的道具的id

注：如果已装备的道具没有满（默认是2个道具槽），targetId和replaceId写成一样，该道具就会装备到空的道具槽

Response：
status返回200 装备成功
无数据
12.获取道具列表 listProperty
Request :

  • userId，整型，用户唯一id

Response：
status返回200 成功
  • equipedProperties ，用户已装备道具数组
示例：
"equipedProperties ": [         //道具数组，玩家已经装备的道具

    },
    ......
]
  • userProperties，用户拥有的未装备道具数组
示例：
"userProperties": [             //道具数组，玩家拥有但没有装备的道具
    {
        id，整型，道具id

        info，道具信息
        level，整型，道具等级
        photo，字符串，道具图片
    },
    ......
]
13.任务推送 taskPush

示例：
"tasks": [             //推送的任务数组
    {
        id，整型，任务id
        title，字符串，任务标题
        score，整型，任务积分
        info，字符串，任务信息
        difficulty，整型，任务难度
    },
    ......
]
14.扫码 scanPrevRule
Request :

  • userId，整型，用户唯一id
  • barCode，字符串，扫码的码值
Response：
status返回400 出错，错误原因查看message

status返回200 规则通过。

15.新任务在线推送接口 newTasks

说明：此接口是服务器主动推送，当系统中有新任务发布时，将新任务推送给玩家的任务任务视图中

Response：

status返回200 

  • type，字符串，任务类型（目前只有"VIEW"，表示任务视图）

  • tasks，任务数组，推送的任务。和第13接口的tasks是一样的

推送事例如下：

{"status":200,"time":1432792948,"message":"有新任务发布了","method":"newTasks","data":{"type":"VIEW","tasks":[{"difficulty":1,"score":10,"lng":116.38914,"title":"新任务推送测试３","taskId":43,"lat":39.92931,"info":"新任务推送测试３\nsb","clues":"新任务推送测试３"}]}}

16.已接受任务消息推送 taskMsg

说明：此接口是服务器主动推送。应用于已接受任务中有前置规则通过的时候的正反馈。

例如：一个线下任务有三个前置规则分别是道具完备性、GPS验证、扫码验证，如果有其中一项或者几项规则验证通过，该规则的正反馈推送给客户端。

Response：

status返回200 

  • taskId，整型，已接受任务的id
  • reply，回复数组
示例：
"reply":[
{"html":"html文本1"},
{"html":"html文本2"},
...
]
